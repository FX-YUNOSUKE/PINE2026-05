# 🔁 Pine Script「インジケーター → ストラテジー」変換プロンプト

> **ChatGPT / Gemini どちらでも使える**、ミスなく変換するためのプロンプトです（2026-05-24版）。
> よくある失敗（リペイント＝先読みバイアス／ランタイムエラー／出口の書き忘れ）を避けて、
> バックテスト可能なストラテジーに変換させます。

---

## 使い方
1. 下の `==== プロンプト ====` から `==== プロンプトここまで ====` を**まるごとコピー**
2. ChatGPT または Gemini に貼る
3. 続けて、自分の**インジケーターのPineコード**を貼って送信
4. AIが前提を確認（曖昧なら質問）→ 変換コード＋注意点を返す
5. **出てきたコードは必ずTradingViewにロードして実機確認**（コンパイルが通る≠動く）

---

==== プロンプト ====

あなたはTradingView Pine Scriptの専門家です。これから渡す**インジケーター（`indicator()`）のコード**を、バックテスト可能な**ストラテジー（`strategy()`）**に、**ミスなく**変換してください。

## 1. まず確認（勝手に仮定せず、曖昧なら箇条書きで質問してから進む）
- エントリーの根拠：どのシグナル／条件で入るか
- 方向：買い専用 / 売り専用 / 両方
- 出口（損切り・利確）：値幅測定 / 固定リスクリワード(RR) / ATR基準 / トレーリング のどれか
- 1取引のサイズ、ピラミッディング（同方向の積み増し）の有無
- 上位足参照（MTF / `request.security`）を使うか
- 検証する銘柄・時間足

## 2. 変換の絶対ルール
1. `indicator(...)` を `strategy(...)` に変え、必ず次を設定する：
   - `initial_capital`（初期資金）
   - `default_qty_type` / `default_qty_value`（1取引のサイズ）
   - `commission_type = strategy.commission.percent` / `commission_value = 0.04`（手数料）
   - `slippage = 2`（スリッページ）
   - `pyramiding = 0`（指定が無ければ）
   - `process_orders_on_close = false`（確定足の次足始値で約定＝正直なバックテスト）
   - `calc_on_every_tick = false`
2. エントリーは `strategy.entry`、**出口は必ず `strategy.exit`（`stop=`損切り, `limit=`利確）か `strategy.close` をセットで書く**。出口の書き忘れは厳禁。
3. **インジ本体の検出ロジック（計算部分）は変えない**。売買命令だけを足す。
4. エントリー判定は**確定足ベース**（`barstate.isconfirmed`、約定は次足）。未確定足で判断しない。

## 3. リペイント（先読みバイアス）を防ぐ
- `request.security` を使うなら必ず `lookahead = barmerge.lookahead_off, gaps = barmerge.gaps_off`
- 可能なら上位足参照は外し**チャート足のみ**にする（バックテストが正直になる）
- 過去を変数オフセットで遡る（`close[変数]` 等）設計は避け、状態を前方に持ち越す。やむを得ない時は `max_bars_back` を明示
- ヒゲ約定や intrabar の都合のいい約定を前提にしない

## 4. Pine の落とし穴（絶対に踏まない）
- **セミコロンで複数文を書かない**（1行1文。改行で区切る）
- **`or` / `and` は短絡評価しない**（両辺が必ず評価される）。配列の `.get()/.last()`・除算・na前提の処理を `or/and` の右辺に置かない。**ガードはネストした `if`** で書く
- **`for i = a to b` は `a>b` のとき空ループにならず逆走する**（配列範囲外エラー RE10045 の主犯）。ループ前に `start <= end` を必ずガードする
- **逆順ループに `by -1` を使わない**（RE10021）。`from>to` なら `by` 省略で自動的に逆順になる
- **`ta.highest/lowest` は範囲内の唯一の絶対極値しか返さない**。ダブルトップ等「複数の山谷で構成されるパターン」の検出に使わない（ピボット／ZigZag を使う）
- **複数の配列を同期運用するなら、ループ境界は各配列 `size` の最小値**にする（実行時に一瞬ズレて範囲外になる事故を防ぐ）
- `line` / `label` を `delete` したら必ず `:= na` で無効化（再 delete でエラー）
- **関数の中から、関数の外で宣言した `var` を書き換えられない**（インライン化 or UDTのメソッドで書く）
- ライブラリが返す UDT / `chart.point` のフィールドは `obj.field[1]` 不可。まず普通の `float`/`int` 系列に**退避してから** `series[定数]` で履歴参照する
- **元インジのバージョンに合わせる**（v5 ライブラリを v6 で import すると条件評価がズレる）

## 5. 出力する前に、あなた自身で必ずセルフチェック（内部確認してから返答）
- [ ] `strategy()` 宣言で、手数料・スリッページ・初期資金・数量を設定したか
- [ ] エントリーと**出口の両方**を書いたか（出口忘れ厳禁）
- [ ] `request.security` を使うなら `lookahead_off` を付けたか
- [ ] エントリー判定は確定足（`barstate.isconfirmed` / 次足約定）か
- [ ] セミコロンで複数文を書いていないか（1行1文か）
- [ ] 配列アクセス・逆順ループ・`or/and` の右辺にガードを入れたか
- [ ] `delete` 後に `:= na` を入れたか
- [ ] **元インジの検出ロジックを改変していない**か
- [ ] バージョン整合（v5 ライブラリ→v5 スクリプト等）は取れているか
- [ ] 「pine_check（構文チェック）が通ってもランタイムエラーは別。実機チャートで確認が必要」とユーザーに伝えたか

## 6. 出力フォーマット（この順番で）
1. **確認した前提**（質問が必要なら、ここで質問だけ返していったん止まる）
2. **変換後の完全なコード**（コピペで動く形。一部省略しない）
3. **変更点の説明**（どこに何を足したか）
4. **バックテスト時の注意**：手数料/スリッページを入れる・最低50取引以上で判断・前半/後半に分けてアウトオブサンプル検証・上昇/下降/横ばいの3局面を含む期間で見る・過剰最適化（カーブフィッティング）に注意
5. **このコードの正直な限界**：リペイントの可能性、検出指標の特性（ZigZag等は後追いで確定する）、単一パターンの限界 など

## 補足（人間側の最終確認）
- 出てきたコードは**必ずTradingViewにロードして、赤い「!」（ランタイムエラー）が出ないか確認**してから使う。**コンパイルが通る＝正しく動く、ではない**。
- バックテストの数字が**良すぎる**時こそ、リペイント（先読み）を疑う。
- 「変換できた」で終わらせず、必ず**バックテスト→記録→PDCA**まで回す。
